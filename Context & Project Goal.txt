Context & Project Goal

The user's ultimate goal is to create a reinforcement learning agent that can play Terraria using a custom interface. Due to the high complexity of the game (continuous action space, sparse rewards, physics stability), the project follows a strict "Crawl, Walk, Run" roadmap utilizing Hierarchical Reinforcement Learning (HRL).



Core Technical Architecture

The project consists of three distinct pillars that must be adhered to:

1. The Game Interface (C# / tModLoader)



Role: Acts as the API and Sensor array.

Input: Receives simplified actions (Move, Jump, Craft(ItemID)).

Output: Sends "God Mode" state data (Grid 20x20, Enemy Lists, Inventory Feature Vectors).

Constraint: Direct memory access is preferred over computer vision. We do not use screen pixels.

2. The Brain (Python / PyTorch / Stable-Baselines3)



Architecture: A "Switchboard" (Manager/Worker) system.

The Manager: A deterministic Python script (State Machine) that analyzes global state (Health, Inventory, Threat Level) and selects the active "Mode."

The Workers: Specialized Neural Networks trained on specific Gyms:

Traversal Agent: Moves from A to B.

Combat Agent: Fights enemies within range.

Extraction Agent: Mines resources.

3. Infrastructure & Scaling Strategy



Current Status: Strict Single-Instance.

Rule: We do not attempt parallel training (multi-env) until the Single Agent system is mathematically stable.

Future: Once stable, we will scale to 8-16 parallel instances at 10x speed.

The "Rules of Engagement" (Niche Constraints)



These are specific technical decisions made to solve RL bottlenecks. We may end up implementing these, but they are good to keep in mind for now:



The Crafting "Cheat" API: The agent does not navigate the UI/Mouse to craft. It sends a Craft(ID) command. The C# mod verifies resources and instantly places the item in inventory.

Item Grouping (Feature Bucketing): We do not feed raw Item IDs (0-5000) to the Neural Network. We feed "Inventory State Vectors" (e.g., BestMeleeDamage, HasLightSource, WoodCount_Normalized).

Smart Cursor: When in Extraction/Mining Mode, the "Smart Cursor" feature is forcefully enabled to reduce the action space complexity.

Housing Validation: We do not use the in-game text query. We use a custom C# helper that returns specific error codes (e.g., Error_MissingLight, Error_RoomTooSmall) to provide clear gradients for learning.

Event Handling: Global events (Blood Moon, Invasion) are treated as high-priority triggers that force the Switchboard into Combat Mode.
